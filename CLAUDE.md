# CLAUDE.md — Autonomous Problem Solver

## What this project is

An autonomous AI agent pipeline that daily selects a real software problem, designs an
architecture, and implements a working MVP — without human involvement. GitHub Actions runs
Steps 1 (problem brainstorm) and 2 (architecture design) on a daily cron; a persistent
Railway worker runs Step 3 (MVP implementation) by polling for new feature branches. All
inter-step coordination happens through GitHub branches and Issues.

---

## Tech stack and key dependencies

- **Language:** Python 3.11
- **AI:** Anthropic Python SDK (`anthropic`) — model `claude-sonnet-4-6`, streaming required for Step 3
- **GitHub API:** `PyGithub` for high-level operations; raw `httpx` for anything PyGithub doesn't expose
- **Env management:** `python-dotenv` (local only; Railway and GitHub Actions use env vars directly)
- **Runtime targets:** GitHub Actions (Steps 1+2), Railway persistent worker (Step 3)

## How to build, run, test

### Local development

```bash
pip install -r requirements.txt
cp .env.example .env
# Fill in ANTHROPIC_API_KEY, GH_PAT, REPO_OWNER, REPO_NAME
```

### Run Steps 1 & 2 locally

```bash
python -c "from actions_runner import run_steps_1_and_2; run_steps_1_and_2()"
```

### Run the Railway worker locally

```bash
python -m worker.main
```

### Run on Railway (production)

Deploy via Railway dashboard or CLI. The `Procfile` defines the worker process.
All env vars must be set in Railway's environment settings — no `.env` file on Railway.

### Testing

There is no automated test suite. Railway is the integration test environment.
Validate changes by deploying to Railway and observing logs for a full poll cycle.
See LESSONS.md: *"Railway is your test environment for anything involving Claude."*

## Conventions

### Naming

- Problem slugs: `YYYY-MM-DD-<five-word-title-in-kebab-case>` — generated by `utils.make_slug()`; never construct slugs manually
- Branch names: always `feature/<slug>` — use `utils.branch_name_for_slug()`
- Issue title prefixes: `[CYCLE]`, `[BLOCKER]`, `[DONE]` — exact strings, matched by regex in `worker/main.py`
- Commit messages follow conventional commits: `feat:`, `fix:`, `docs:`, `chore:`

### Error handling

- Never raise from poll loop handlers — log and continue; the loop must not crash
- GitHub API errors: log with full context (slug, operation, status code), then return gracefully
- Claude API errors: `ClaudeClient.complete()` has built-in retry logic — do not add extra retry wrapping on top
- All terminal state changes (DONE, BLOCKER, CANCELLED) must be attempted even if a preceding step raised

### Logging

- Use `logger.info()` at every state transition: branch qualified, Claude call start, `stop_reason`, file committed, label changed, issue number
- Log `stop_reason` on every Claude response — it is load-bearing (`max_tokens` means truncation, not completion)
- Log slug and issue number together wherever possible — makes log triage fast

### Commit discipline

- One logical change per commit; reference the GitHub Issue number in the closing comment
- Validate on Railway before closing an Issue
- Link commit hashes in Issue closing comments

## What NOT to do

### Never touch these without understanding the full lifecycle state machine

- **`worker/main.py` — `processed_fresh`, `processed_resumed`, `processed_cancelled`**: These in-memory sets are performance optimisations only. They are not the source of truth. GitHub branch/issue state is. Do not add correctness logic that depends on these sets surviving a container restart.
- **`worker/step3.py` — `_handle_truncation()`**: The `processed_resumed.discard(slug)` call after adding `cycle-resume` is intentional and required. Removing it silently breaks auto-resume.
- **`shared/markers.py`**: `DONE.md` and `CANCELLED.md` are terminal markers that the poll loop checks for. Do not rename, relocate, or conditionally skip writing them.

### Never do these in the GitHub client

- **Do not call `read_file()` without stripping base64 newlines**: GitHub paginates base64 with embedded `\n`. Always use `.replace("\n", "")` before `base64.b64decode()` — see the correct pattern in `load_all_problem_mds()`. Bug tracked in issue #53.
- **Do not add new issue-label operations without calling `_ensure_label()` first**: Labels must exist before they can be applied; PyGithub will raise if the label doesn't exist.
- **Do not use `repo.get_issues()` without a state filter**: This returns all issues and can be slow or paginated unexpectedly; always pass `state="open"` or `state="closed"` explicitly.

### Never do these with Claude prompts

- **Do not name prompt sections `ARTIFACT 1:` / `ARTIFACT 2:` or similar**: Label names leak into Claude's output and get committed to files. Use positional instructions instead.
- **Do not call `claude.complete()` with `max_tokens > ~8192` without `use_streaming=True`**: The Anthropic SDK raises a `ValueError` for non-streaming requests that may exceed 10 minutes. Step 3 uses `max_tokens=32768` and requires streaming.
- **Do not use `use_web_search=True` together with `use_streaming=True`**: Web search tool calls are handled inside the SDK's agentic loop; streaming and tool use are incompatible in the current client implementation.
- **Do not modify `STEP3_SYSTEM` without re-reading LESSONS.md first**: The terminal block instructions (`<<<MVP_COMPLETE>>>`, `<<<BLOCKER>>>`) and anti-preamble rules were tuned across many failed runs. Prompt changes here are high-risk.

### Branch and PR policy

MVPs are self-contained projects with no relation to each other.
`main` contains only agent infrastructure — never MVP code.
When Step 3 completes, the PR is **closed without merging** (Policy B).
The `done` label and `DONE.md` on the feature branch are the completion signal.

Do not merge feature branches into main.
Do not suggest merging a done PR as a cleanup or housekeeping action.

### Never do these in the worker

- **Do not store correctness-critical state in Python variables**: Container restarts on Railway reset all in-memory state silently. Any state that must survive a restart (e.g. resume count) must be written to the feature branch as a file.
- **Do not close a `[CYCLE]` issue programmatically**: The worker only updates labels on CYCLE issues; it never closes them. Closing is a human action.
- **Do not skip the `RESUME_COUNT` check in `_qualifies_for_step3()`**: This guard prevents already-attempted branches from being re-picked up as "fresh" after a container restart. Removing it causes duplicate Step 3 runs.

### Do not create `docs/` files

There is no separate specification document. The authoritative description
of the system is the code itself, CLAUDE.md, and LESSONS.md.
Do not create new documents in a docs/ directory.

## Operational log archive

Downloaded Railway deploy logs and GitHub Actions run logs are stored locally in:

```
logs/
  railway/    # Railway worker deploy and runtime logs
  actions/    # GitHub Actions workflow run logs
```

These files are gitignored — they never appear in commits or git history.
**Do not read, analyse, or reference files in `logs/` unless the user explicitly asks.**
They are passive archives, not inputs to any coding task.
